# 飞书多维表格数据源连接器 — 开发标准规范

本规范旨在约束“抖音罗盘与资金明细飞书连接器”系统的设计与后续迭代，确保系统在高并发多维表格同步场景下的**高稳定性**、**可追溯性**、**智能容错预警**以及在**宝塔面板（BT Panel）环境下的轻量独立运行**。

---

## 1. 客服与联系人链接配置化管理标准

### 1.1 背景与痛点
系统界面包含“联系客服”或“添加联系人”的导流入口。若将其硬编码在前端代码或静态 JSON 中，当邀请链接由于飞书安全时效过期或客服人员变动时，需要重新编译前端包并重新发布，极大增加了运维成本。

### 1.2 开发实施标准
* **动态存储**：所有的客服加人链接必须持久化存储在数据库全局配置表（如 `system_configs`）中。
  * 默认存储链接：`https://www.feishu.cn/invitation/page/add_contact/?token=0das1471-15f8-4e96-b73b-9fc5ab63de7e`
* **API 穿透获取**：
  * 后端提供轻量 API `GET /api/v1/system/configs` 获取当前系统配置，其中包含 `customer_service_url` 参数。
  * 前端组件在初始化时异步拉取此接口，若接口请求失败或数据为空，则采用预留的兜底配置展示。
* **安全性防范**：存储的链接应在入库时进行基础的安全字符校验，防止恶意脚本注入。

## 2. 数据库持久化存储与功能分布标准

### 2.1 存储原则
连接器不仅仅是网络请求的中转站，更需要将核心的抓取记录、账号凭据状态进行本地持久化，实现与插件管理后台以及其他分析系统的无缝接入。

### 2.2 开发实施标准
* **多系统对接预留**：所有抓取到的真实订单数据、资金流水等，除了直接响应回传给飞书外，必须以去重、归档的方式保存至本地数据表中，为后续管理后台与大屏分析工具提供原始数据依托。
* **数据去重与索引优化**：以第三方平台（如抖音侧）的唯一主键（如 `flow_id` 或 `order_no`）作为数据库的 `UNIQUE` 唯一约束，确保重复拉取时不会造成数据翻倍膨胀，实现“覆写/无损去重”入库。
* **文档说明书输出**：必须在项目最外层输出并维护 [数据库以及功能分布.md](file:///Users/wangxun/Desktop/demo/26xm/飞书连接器规划/数据库以及功能分布.md)，清晰注明每个表的建表 Schema、字段说明、索引规划，以及功能模块在各表之间的读写逻辑。

## 3. 企业自建应用接口报错预警与推送机制

### 3.1 预警目标
当连接器在后台运行定时增量同步或被飞书引擎调用时，若遭遇 **Cookie 过期 (CredentialsExpired)**、**抖音接口 500 报错** 或 **网络严重超时** 等异常，系统需第一时间主动推送告警给管理员（王某），便于人工介入，避免数据同步长时间中断。

### 3.2 开发实施标准
* **飞书推送链路**：
  * 后端捕获到严重 `Error` 时，通过内置的飞书企业自建应用 API，向管理员发送飞书即时消息。
  * 接口流程：
    1. 使用自建应用的 `app_id` 与 `app_secret` 向飞书获取 `tenant_access_token`。
    2. 调用飞书消息发送接口 `POST /open-apis/im/v1/messages?receive_id_type=open_id`。
    3. 向管理员用户的 `open_id`（或通过 Webhook 群机器人）推送富文本卡片消息。
* **告警限频与避峰机制 (Anti-Spam)**：
  * **痛点**：若同步任务为高频轮询，一旦 Cookie 失效，系统会因频繁报错而疯狂向管理员推送消息，造成“消息炸群”。
  * **解决方案**：引入基于内存/SQLite 锁的**报错频控限制器**（如 Token Bucket 或滑动窗口）。**同类型的异常（如 Cookie 过期）在 1 小时内仅允许向管理员推送一次**，后续相同报错在期间仅进行本地静默日志归档，不再重复报警。
* **消息卡片（Interactive Card）规范**：
  推送的消息必须采用卡片形式，包含以下要素：
  1. **状态头**：🚨 飞书连接器同步异常预警 (红色主题)。
  2. **故障内容**：同步模块（如“账户中心”）、报错原因（如“Cookie已失效，请重新绑定”）。
  3. **时间戳**：精准到秒的报错发生时间。
  4. **排障操作**：卡片底部附带“一键前往配置页面”的快捷跳转链接。

## 4. 稳定性、宝塔部署与资源占用指标规范

### 4.1 宝塔面板 (BT Panel) 部署隔离标准
连接器需要在宝塔 Linux 面板的 Node 项目管理器中运行，必须做到“零资源泄漏、独立端口隔离”，绝不能干扰宝塔服务器上的其他项目。

### 4.2 开发实施标准
* **防死机/防崩溃机制**：
  * 严禁在异步操作中出现未捕获的全局异常。必须在 Express 入口注册全局异常捕获中间件：
    ```javascript
    process.on('uncaughtException', (err) => {
      console.error('未捕获的全局异常:', err);
      // 记录本地日志并尝试发送飞书预警
    });
    process.on('unhandledRejection', (reason, promise) => {
      console.error('未处理的 Promise Rejection:', reason);
    });
    ```
  * 当同步接口遇到严重底层错误时，应向飞书多维表格引擎返回符合协议的 JSON 报错结构（包含 `code: 500`, `message: "描述"`），**而不允许直接导致 Express 监听线程挂掉/死机**。
* **进程托管运行**：
  * 项目部署必须采用 `PM2` 进程管理工具进行托管。
  * 限制单实例的内存阈值，配置 `ecosystem.config.js` 包含 `max_memory_restart: 150M`，当发现内存由于 GC 延迟等原因超出 150MB 时，由 PM2 自动执行零停机热重启。
* **资源占用限制指标**：
  * **内存驻留 (RSS)**：
    * 待机闲置状态下：内存占用需控制在 **`50MB ~ 80MB`** 之间。
    * 数据拉取与 SQLite 批量入库状态下：内存峰值需控制在 **`120MB`** 以内。
  * **CPU 占用**：
    * 闲置保活状态下：单核 CPU 占用率小于 **`1%`**。
    * 极速同步数据拉取状态下（500 条大页宽多线程映射清洗）：单核 CPU 瞬时占比控制在 **`15% ~ 30%`**，且高频段持续时间不得超过 3 秒，防止 CPU 吃满影响同服务器上的其他 PHP/Java 站点。
  * **日志膨胀限制**：
    * 定时同步的控制台日志需对接宝塔日志切割系统，或在后端配置按天自动滚动的日志器（如 `winston-daily-rotate-file`），单个日志文件超出 10MB 时自动切分，本地仅保留最近 7 天的日志，防止塞满宝塔系统盘。

## 5. 高并发性能测试与响应生成速度规范

### 5.1 高并发测试要求
为了保证在飞书多维表格多用户同时操作或大批量定时自动同步并发涌入时，连接器服务不会挂起或死机，系统必须在上线前通过**高并发压力测试**：
* **高并发测试区间**：系统必须通过至少 **50 次以上并发请求（>= 50 Concurrent Requests）**的并发压测。要求在并发压测下，不出现连接重置、请求超时丢失、线程死锁或 HTTP 502/504 等现象。
* **数据库并发隔离**：针对 SQLite 单文件数据库多并发写入易触发 `SQLITE_BUSY`（数据库锁死）的局限，后端必须启用 SQLite 的 **WAL (Write-Ahead Logging) 写入日志模式**，或引入内存队列写机制，保障高并发读写的稳定。

### 5.2 数据生成与响应速度要求
* **响应时延 (Latency) 指标**：在排除掉为了防范抖音风控而人为加入的随机休眠延迟（Delay）之后，连接器对单次请求的数据清洗、结构映射与 JSON 生成的**纯计算响应耗时 (TTFB) 必须控制在 500ms 以内**。
* **大页宽转换效率**：当单页返回 500 条大页宽数据时，后端在内存中的转换和回传序列化必须高效，避免阻塞 Node.js 事件循环主线程，以此保证对外部请求的极速响应能力。

## 6. 增量同步与接口防封控开发规范

### 6.1 避免全量同步原则
抖音以及各主流电商平台的私密接口对调频有严格的风控墙。**严禁系统在每一次执行同步动作时，都去请求和回传几千条全量的历史记录**。这会导致服务器频控被拉黑或 Cookie 迅速失效。系统必须支持并默认采用增量同步模式。

### 6.2 增量同步实施标准
* **滑动时间窗口截断**：
  * 后端接口必须能够解析飞书发来的上次同步时间戳（`Last Sync Time`），或读取本地数据库上一次同步的最新动账时间。
  * 仅以该时间点向前稍微偏移一段安全滑动缓冲窗口（如前推 1 小时或 T-1 天，防止因单据延迟产生漏单）作为抖音接口的 `start_time` 进行请求。在源头上拒绝全量拉取。
* **本地主键碰撞与增量推送**：
  * 在向飞书多维表格引擎回传记录前，将最新获取的数据与本地 SQLite 表（如将来的归档流水表）中已存在的 `flow_id` 唯一主键进行碰撞对比。
  * **仅将本次请求中发现的、未曾同步过的新增动账记录或者发生状态变更的异常记录回传给飞书**。对于无变化的历史数据直接做过滤，使数据包极小化，从而大幅节省服务器带宽，并彻底降低飞书多维表格更新时的 DOM 渲染卡顿。

## 7. 接口安全签名校验与大文件解析防爆规范 (Network & Upload)

### 7.1 安全验证要求
外部连接器暴露 the API（尤其是元数据获取和同步拉取接口）需要防止越权和重放攻击。
* **飞书协议签名强校验**：
  * 生产环境下，后端对于 `/api/table_meta` 和 `/api/records` 这类飞书引擎直调的端口，必须**强制执行飞书连接器签名验证**：
    1. 校验请求头中的 `x-base-signature` 签名是否合法。
    2. 校验 `x-base-request-timestamp` 请求时间戳，对于超过 5 分钟的请求直接拒绝，防止网络重放攻击。

### 7.2 文件导入内存防爆要求
针对业务人员手工上传 Excel/CSV 或 RPA 定时通过 Webhook 上传大量导出的账单明细时，系统需要具备防 OOM 崩溃能力：
* **上传大小上限限制**：严格限制文件上传网关的 Request Body 限制为 **`5MB`** 以内，禁止任何超大文件入库。
* **流式读取与垃圾回收**：
  * 数据解析引擎必须采用 **Stream 流式读取解析器**（如 `xlsx-stream-reader` 或 `csv-parser`），禁止一次性将大型 Excel 文件完整加载至堆内存进行同步遍历，确保在宝塔多站点服务器部署环境下不因单个用户的解析导致内存泄漏或崩溃。

## 8. 模式 B 模拟登录下防爬虫检测与并发控制规范 (Anti-Crawler & Jitter Delay)

### 8.1 防爬虫防风控背景
模式 B 使用截获的 Cookie 和模拟浏览器行为来获取数据，若接口请求频率过快、并发过多或规律性过强，极易触发抖音等电商平台的反爬虫（Anti-Bot）安全策略，导致商户账号被风控、IP 遭拉黑或 Cookie 快速过期失效。

### 8.2 开发实施标准
* **随机休眠延迟 (Jitter Delay) 机制**：
  * **实施标准**：后端在发起真实的网页数据请求（如分页拉取）时，**单次 HTTP 请求之间必须强制加入 1500 毫秒至 3000 毫秒的随机休眠延迟（抖动延迟 Jitter）**。
  * **逻辑实现**：
    ```javascript
    const delayMs = Math.floor(Math.random() * 1500) + 1500;
    await new Promise(resolve => setTimeout(resolve, delayMs));
    ```
  * **目的**：打碎固定的周期性请求节奏，模拟人工点击操作，避开针对高频固定请求间隔的防爬虫审计。
* **商户级串行同步控制 (Queueing per Tenant)**：
  * **实施标准**：针对同一个店铺/商户，**其多页翻页拉取或多模块同步任务必须严格采用串行队列机制进行**（同一时间只能有一个同步线程在该 Cookie 下运行），禁止多页并发并行请求，降低单个 IP 对相同接口的瞬时并发数。
* **指纹仿冒与 Header 对齐**：
  * 网络请求头必须携带与登录环境设备指纹一致的 `User-Agent`，并且必须包含必要的 `Referer`、`Origin` 与 `Content-Type: application/x-www-form-urlencoded` 表单特征，防止被抖音网关由于“Header 缺失”被判定为非浏览器机器人进行阻断。
